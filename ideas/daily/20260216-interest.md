# 興味関心メモ: 2026-02-16

## 1. AIエージェントの「枠組み戦争」 — Skills標準化・ACP・klaw.shが示すオーケストレーション層の急成長

はてブ「Skills機能比較まとめ」(#1, 290 users)、AWS Blog「Kiro ACP対応」(#20)、HN「klaw.sh」(#12, 21pt)が同時に登場。**AIエージェントの「中間層」— スキル配布・エージェント間通信・オーケストレーション — が急速に標準化・プロダクト化している**。

### Claude Code / Codex / Gemini CLI — Skills機能の3社比較（290 users）
- **共通基盤**: 3ツールとも`SKILL.md`ファイルを中心とした**Agent Skills標準**に準拠
- **Claude Code**: `~/.claude/skills/`配置、`/skill-name`スラッシュコマンド呼出、`context: fork`でサブエージェント連携、**Progressive Disclosureでコンテキスト節約**
- **Codex**: `$skill-installer`でリモートリポジトリからインストール可能、**AGENTS.mdとの階層的共存**、ディレクトリ単位のoverride.md
- **Gemini CLI**: UIによる明示的な**同意モデル（セキュリティ重視）**、`/skills`コマンドで有効/無効管理、Extensionシステム配布
- **使い分けの核心**: コンテキストファイル（CLAUDE.md等）はプロジェクト全体ルール、スキルはオンデマンド専門タスク
- **所感**: 自分のskills構成（`neta-trend-daily`等）はClaude Code固有だが、**標準化が進めばツール間ポータビリティ**が生まれる可能性

### Kiro ACP対応 — エージェント間通信プロトコルの標準化
- **ACP（Agent Communication Protocol）**: AIエージェント同士が標準化されたプロトコルで通信する仕組み
- **Kiro 0.9の同時発表**: カスタムサブエージェント + オープンウェイトモデル + ACP対応の3点セット
- **意味**: MCPが「AIとツール」の接続を標準化したように、**ACPは「AIとAI」の接続を標準化**する
- 02/13のinterestメモ(#2)で取り上げた**MCPの二方向進化（サーバーサイド＋WebMCP）に続く第3の標準化層**

### klaw.sh — AIエージェント用Kubernetes（21pt）
- **コンセプト**: Kubernetesの知識をそのままAIエージェント管理に転用
  - Pod → エージェントインスタンス、Namespace → ワークスペース隔離、ConfigMap → SOUL.md設定、CronJob → klaw cron
- **300以上のモデル対応**、単一バイナリ、分散実行可能
- **組み込みツール**: bash, ファイル操作, 検索, Web, **サブエージェント生成**
- **スケジューリング**: Cronスタイルの反復タスク自動化 — 自分の日次トレンド収集のような定期タスクに適合
- **マルチチャネル**: CLI, TUI, Slack, API
- **所感**: 今の自分のワークフロー（Claude Code + GitHub Actions）をより体系的に管理できる可能性。特に**マルチエージェント構成の本番運用**に有用

### 02/13との接続 — 「ハーネスエンジニアリング」の制度化
- 02/13で「ハーネス（モデルをどう使うかの設計）が性能を決める」と分析した
- 今日の3記事は、そのハーネスが**個人の工夫→業界標準→プロダクト**へと昇華する過程を示している
- Skills標準 = ハーネスの共有形式、ACP = ハーネス間の接続規格、klaw.sh = ハーネスの運用基盤

## 2. AGENTS.mdの自動育成 — エージェント向けドキュメントの「20行制約」

はてブ「AGENTS.mdを自動で育てる仕組みを作った」(#2, 271 users)。**271ブクマという高反応は、エージェント向けドキュメント管理が多くの開発者の課題である**ことを証明。

### 核心的な問題認識
- **LLMの命令追従能力には上限がある**: 150〜200個程度の指示で飽和し、冗長な指示はエージェント性能を**全体的に低下**させる
- **「20〜30行のバジェット制約」を守ることが最優先** — これは意外と知られていない重要な知見

### 記載すべき/すべきでないの明確な線引き
- **記載すべき**: コード推測不可の固有判断、非自明なビルド/テストコマンド、重要な落とし穴
- **記載不可**: リンターに任せるコード規約、変動するディレクトリ構造、汎用的なプログラミング助言
- **設計原理**: 「エージェントが自力で推測できることは書かない」

### 実装の工夫
- **Git post-checkout hook + template directory**で`git clone`時に自動生成
- ルートディレクトリ直下3項目未満 → 空リポジトリと判定 → AGENTS.md生成対象
- **HTMLコメントで「構造変更禁止」「削除禁止」を埋込** — エージェントによる自己改変を防ぐ
- テンプレート内に「プレースホルダー定義」を具体化し、自己参照削除を防止

### 自分のワークフローへの示唆
- 現在の`.claude/rules/user-profile.md`や`.claude/skills/`は**バジェット制約を意識していない**
- 特にuser-profile.mdは興味領域の列挙が長くなりがち → **エージェント性能に悪影響の可能性**
- **次のアクション**: CLAUDE.mdとskills定義のトークン数を測定し、20-30行制約に収まっているか検証する

## 3. OpenClawサプライチェーン攻撃 — AIエージェントエコシステムの「npm left-pad問題」

はてブ「AIエージェントの12%がマルウェアだった」(#5, 130 users)とReddit「OpenAIがOpenClaw買収」(#34, 506 ups)。**同じ日にセキュリティ問題と買収ニュースが並ぶ**のは、エージェントエコシステムの急成長とリスクの両面を象徴。

### 攻撃の全貌
- **規模**: ClawHubマーケットプレイスの2,857スキル中、**341個（約12%）がマルウェア**
- **確率**: 8個インストールしたら1個がマルウェア — 壊滅的な汚染率
- **攻撃手法**: 正規ツールに偽装し「前提条件」セクションで外部ファイルダウンロードを誘導 → **AMOS Stealerマルウェア**をインストール
- **窃取対象**: ウォレット秘密鍵、取引所APIキー、SSH認証情報
- **CVE-2026-25253**: 悪意あるWebページ訪問だけで完全乗っ取り可能。WebSocket検証の欠如を悪用

### 被害と対策
- **21,639台のOpenClawインスタンスがインターネット公開状態**（Censys調査）
- 推奨: インストール済みスキル即確認、Ciscoの「Skill Scanner」活用、C&Cサーバー通信ブロック

### トピック#1との接続 — Skills標準化のセキュリティ側面
- Claude Code / Codex / Gemini CLIの3社比較(#1)で、**Gemini CLIが「同意モデル（セキュリティ重視）」**を採用しているのは、まさにこのリスクへの対応
- **Codexの`$skill-installer`（リモートリポジトリからインストール）は、OpenClawと同じ攻撃面を持つ**
- Claude Codeのskillsはローカルファイルベース → 相対的に安全だが、将来的にマーケットプレイスが出来た場合は同じリスク
- **「スキルのサプライチェーンセキュリティ」は、npm/PyPIの教訓を活かすべき領域**

## 4. AIバンパイア問題 — 生産性向上の果実は誰のものか

Simon Willison「AIバンパイア」(#22)とr/ClaudeCode「2026年もAIは開発者を置き換えられない理由」(#28, 183 ups/259 comments)、はてブ「AIコーディングが隆盛する一方でマネジメントが辛い件」(#9, 17 users)。**AI活用の「光と影」が3つの異なる視点から同時に浮上**。

### AIバンパイア — Steve Yeggeの警告
- **核心**: AIを活用して生産性を10倍にしても、その価値のほぼ100%を企業が吸収する。開発者には何も残らない
- **認知負荷の問題**: エージェント作業は**数時間の短い期間のみが現実的**。練習を積んでも限界がある
- **構造的な問題**: AIで簡単な作業が自動化 → **複雑な意思決定が開発者に集中** → 精神的負担増大 → バーンアウトリスク上昇
- **Deep Work思想との直結**: 「AIが定型作業を処理する分、人間は深く考える時間を確保すべき」— しかし現実には**「深く考える」ことばかりが残り、休息のない状態**に

### 2026年もAIは開発者を置き換えられない理由（183 ups / 259 comments）
- **259コメントの大議論** — AI時代の開発者の存在意義への切実な問いかけ
- 02/13のinterestメモ(#3)で取り上げた「シニアSWE、まだコードを書いていますか？」（134 ups/178 comments）の続編的な議論
- **両記事の共通テーマ**: AIはコード生成はできても、**仕様の曖昧さの解消・アーキテクチャ判断・ビジネスコンテキストの理解**は人間が必要

### AIコーディング時代のマネジメントの苦悩（17 users）
- **著者の立場**: 開発組織でAI導入を推進するマネージャー
- **具体的な葛藤**:
  - MCPサーバー導入管理: 任意パッケージ許可 → サプライチェーン攻撃リスク（トピック#3と直結）
  - IDE統一・拡張機能ホワイトリスト化 → メンバーの自由制限への反発
  - AIコードレビュー: 「人間が理解していないコード」の混入リスク（パスワードをクエリパラメータに載せる例）
- **心理的課題**: 自分が「コンサバティブな立場」に追いやられる違和感、メンバーへの公平評価への自信喪失
- **所感**: **セキュリティ重視のマネジメントは正当だが、メンバーの「AI活用意欲」を殺さないバランスが極めて難しい**

### 3つの視点の統合
- **Yegge（個人）**: AI活用の果実が個人に還元されない構造問題
- **r/ClaudeCode（コミュニティ）**: AI vs 人間の役割分担の模索
- **Zenn（組織）**: AI導入のセキュリティ vs 自由のバランス
- **共通の教訓**: AIツールの導入は技術問題ではなく、**組織設計・報酬設計・信頼設計の問題**

## 5. 本日のメタ分析 — エージェントエコシステムの「インフラ化」

今日の4トピックを俯瞰すると、**AIエージェントが「ツール」から「インフラ」へ移行する過渡期**が見えてくる。

### パターン: ツール → 標準 → インフラ → セキュリティ問題
1. **ツール段階**: 各社がスキル機能を独自実装（Claude Code Skills, Codex Skills, Gemini Skills）
2. **標準化段階**: Agent Skills標準、ACP（Agent Communication Protocol）の登場
3. **インフラ化段階**: klaw.sh（Kubernetesモデル）、Kiro 0.9（サブエージェント＋エンタープライズコントロール）
4. **セキュリティ問題段階**: OpenClawの12%マルウェア — npmやPyPIが辿った道と同じ

### 02/13からの進化
- 02/13: 「ハーネスエンジニアリング」= 個人がモデルを制御する技術
- 02/16: そのハーネスが**業界標準・運用基盤・セキュリティ体制**を必要とする段階に到達
- **AGENTS.mdの「20行制約」**は、インフラ化に伴う複雑性への対抗策

### 自分への示唆
- **短期**: CLAUDE.md/skills定義のバジェット制約を検証（トピック#2のアクション）
- **中期**: klaw.shやACP対応の動向を追い、マルチエージェント構成の本番運用を検討
- **長期**: スキルのサプライチェーンセキュリティは、エコシステム参加者全員の課題。OSS開発への関心とも接続
