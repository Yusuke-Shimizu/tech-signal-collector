# 興味関心メモ: 2026-03-04

## 1. Claude Code Agent Teams — マルチエージェント協調の実戦投入

3/3メモでは「2エージェント体制（spec-author + spec-critic）」「4ドキュメント体制」が確認された。今日は**Agent Teamsという本格的なマルチエージェント協調機能**と、**Gitをベースにした第二の脳の自律運用**という2つの事例から、Claude Codeの活用が個人から組織レベルへ飛躍するフェーズに入ったことが見える。

### [Claude Code Agent Teamsの衝撃と実際](https://gihyo.jp/article/2026/02/get-started-claude-code-07)（130 users）

- **Agent Teamsの核心**: Sub Agents（メインに報告）と異なり、Teamsは「チームメンバー同士が直接P2Pで通信」しながら共有タスクリスト（`~/.claude/tasks/[team-name]`のJSON）で協調
- **起動**: `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` + プロンプトに「チームで行ってください」
- **表示モード**: in-process（単一画面切り替え）/ tmux（分割ペアン同時表示）/ iTerm2（Mac向け）
- **コスト警告**: 標準エージェント比で**トークン消費が約7倍**。Sonnet/HaikuをTeammate、Opusをリードに割り当てるコスト最適化が必須
- **最適ユースケース**: 大規模リファクタリング、フロントエンド/バックエンド並列開発、マルチアングルのコードレビュー・バグ調査
- **限界**: プレビュー機能のため、Rewind/Resumeが不可。方向性の悪いTeamsは修正コストが高い

### [Claude Codeで「第二の脳」を作り、最終的にAIが自律運用するようになった話](https://qiita.com/yamapiiii/items/cc2450f410b64329d275)（78 users）

- **Git選択の決定的理由**: Claude CodeがAPI不要でファイルシステムを直接操作できる点 → Notion/Obsidianより圧倒的に相性が良い
- **情報フロー設計**: `00_Inbox → 10_Journal/20_Projects → 30_Tech_Notes → 99_Archives` の番号付きフォルダ構造。物理的な構造がAIの判断基準になる
- **Brain-Project分離の原則**: Brain（Why/What: 目的・仕様・意思決定ログ）/ Project（How: コード・テスト・設定）の明確な分離でドリフト防止
- **3層自動化**: カスタムコマンド14個（`/decompose`/`/work`/`/orchestrate`等）→ PM自動介入層（優先度スコアリング・停滞パターン診断）→ GitHub Actions（毎朝Inbox整理）
- **CLAUDE.mdの核心設定**: 「確認なし即実行・やってから報告」「Brainに記録されていない情報は未記録と返す」
- **本質的な洞察**: **「必要な情報に適切にアクセスできるAI」こそが優秀なエージェント**。全知全能のAIより、情報フローの設計が重要

### 3/3との接続

- 3/3: PdMによる「2エージェント体制（spec-author/spec-critic）」「4ドキュメント体制（PLAN/SPEC/TODO/KNOWLEDGE）」
- 3/4: Agent Teams（P2P協調）+ Git知識ベース（自律運用）→ **個人の自律運用から、チーム規模の並列エージェント協調へ**
- 自分のtool: tech-signal-collectorにも「収集エージェント / スコアリングエージェント / 記事深掘りエージェント」のTeams構成を試す価値がある

## 2. Claude Code安定活用の7鉄則 — 実践知の結晶

### [Claude Codeベストプラクティス -- 成果を安定させる7つの鉄則](https://qiita.com/nogataka/items/392934f79e943e8b9228)（55 users）

実務で使い込んだ著者による「成果を安定させる」ための体系的なまとめ。

**基本動作**:
- **鉄則1（具体的に指示）**: 対象ファイル・修正箇所・目的・完了条件・制約の5項目を含めることでターン数削減
- **鉄則2（必ず検証）**: diff確認・テスト実行・コスト確認の3ステップ。「動いたから大丈夫」はエッジケース見落としのリスク

**環境整備**:
- **鉄則3（CLAUDE.md）**: 繰り返し指示（「TypeScriptで書いて」「テストはVitestで」）をCLAUDE.mdに集約。手書きで丁寧に書くことが重要（AI自動生成は精度低下のリスク: 記事4参照）
- **鉄則4（セッション管理）**: 1タスク1セッション。長時間セッションは出力品質が低下。`/compact`で履歴要約

**ワークフロー**:
- **鉄則5（計画→実行）**: 手作業5分以上のタスクは計画を提示させてからレビュー・実行
- **鉄則6（コスト意識）**: `/cost`で消費確認。指示の具体性・不要ファイル除外・セッション分割でコスト抑制

**自動化**:
- **鉄則7（自動化の入口）**: 1〜6習得後にHooks・MCPサーバー・Agent Teamsへ進む。段階的な習熟が重要

**実践上のポイント**: 鉄則1と2から始めれば十分効果が出る。全鉄則を一度に導入する必要なし

## 3. Claudeのデータ管理とエコシステム拡大 — メモリ管理とChatGPT抜き首位

### [Claudeのメモリインポート機能](https://simonwillison.net/2026/Mar/1/claude-import-memory/)（Simon Willison / 2026-03-01）

- **機能**: `claude.com/import-memory` でClaudeのユーザー設定をエクスポート/インポート
- **エクスポート対象**: 応答スタイル指示（トーン・フォーマット）/ 個人情報（名前・所在地・職業・関心事）/ プロジェクト情報（目標・定期的トピック）/ 技術環境（ツール・言語・フレームワーク）/ 行動修正設定
- **設計思想**: 「要約や省略なし」の完全なリスト出力 → **AIデータポータビリティと透明性**を重視した設計
- **実用的な価値**: 自分のClaude設定（ユーザープロファイルに近い情報）をエクスポートして他のサービスや将来のClaude移行時に活用できる

### [アンソロピック「前例ない需要」でAIに障害、ChatGPT抜き首位に](https://www.nikkei.com/article/DGXZQOGN02B0X0S6A300C2000000/)（262 users）

- **事実**: 2026年3月2日、Claudeがサービス障害。米国の無料アプリランキングでChatGPTを抜き**1位**に
- **需要急増の要因**: (1) コーディング自動化能力の高さ (2) AI安全方針・軍事利用拒否というブランドポジション
- **背景**: DOD（米国防総省）との緊張関係 — 軍事AI利用への慎重な立場が一部ユーザー層の信頼を強化
- **市場インプリケーション**: Claude Code Agent Teams（記事1）/ 第二の脳（記事2）のような高度な活用事例の普及が需要急増を牽引している可能性が高い

## 4. AI生成コードの認知負債 — インタラクティブ可視化による解消

### [インタラクティブ解説: エージェント型エンジニアリングパターン](https://simonwillison.net/guides/agentic-engineering-patterns/interactive-explanations/)（Simon Willison / 2026-02-28）

3/3メモ#5の深掘り。3/3では「概念の紹介」だったが、今日は実装手法の詳細を記録。

- **認知負債の定義**: エージェントが書いたコードの動作原理を理解できなくなること。新機能計画を阻害し、技術負債と同様に進捗を遅滞させる
- **解消法**: LLMに「インタラクティブなHTML可視化ツール」を生成させる
  - 実例: Rust製の単語雲生成ツールのArchimedean螺旋配置アルゴリズム → 単語配置の探索試行をリアルタイム描画するHTMLをエージェントに生成させた
  - スライダー操作でフレーム単位制御など、インタラクティブ要素で理解を大幅に向上
- **2段階プロセス**: (1) 線形ウォークスルー（コード構造の把握・文書化）→ (2) アニメーション説明（アルゴリズム動作の時系列可視化）
- **適用範囲**: 非自明なアルゴリズム、複雑なデータ変換、並列処理フロー

**自分への示唆（3/3からの継続）**:
- tech-signal-collectorの記事収集→スコアリング→★★★分類パイプラインをClaudeにインタラクティブHTMLとして可視化させる
- 特に「★★★判定のスコアリングロジック」を可視化すれば、パイプライン改善時の認知負債を削減できる
- Agent Teamsを使う場合、各エージェントの判断フローも同様に可視化する価値がある
