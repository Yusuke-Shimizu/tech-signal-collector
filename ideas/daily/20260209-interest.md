## 1. Claude Code Agent Teams — 実践知が一気に蓄積された日

はてブでAgent Teams関連記事が5本同時にホットエントリ入り（合計407 users）。r/ClaudeCodeでもセットアップガイド(334 ups)が話題。**機能発表から実践フェーズに移行した転換点**。

### ロールプレイ駆動開発（98 users）
- エージェントにキャラクター性（涼宮ハルヒのキャラ等）を持たせて開発する手法
- キャラの性格がコード品質や提案方向に影響 — 単なるお遊びではなく**多視点レビューの仕組み**として機能
- 「実務的であることも大切だが、楽しさも大事にしていこう」という開発体験重視の姿勢
- **ポイント**: Agent Teamsの差別化は「エージェント間の直接コミュニケーション」にある。ロールプレイはその特性を活かす面白いアプローチ

### オーケストレーション実践 — Next.jsブログ開発チーム（192 users）
- フロントエンド/バックエンド/DB/QA/UI・UXの5役割でチーム構成
- 表示モード: in-process（単一ターミナル）とSplit Pane（分割表示）の2種類
- タスク進捗は `~/.claude/teams/` にJSON形式で記録
- **実測での知見**:
  - メンバー1人あたり5〜6タスクが最適バランス
  - ファイル競合リスク → 役割の重複回避が必須
  - 承認ボトルネック → 各メンバーの操作承認が頻発して人間が待たされる
  - トークン消費が通常セッションの数倍
- **結論**: 複雑プロジェクトには有効、単純タスクには過剰。タスク規模設計と役割分離が成功の鍵

## 3. AIの生産性パラドックス — 複数ソースが同じ結論に

はてブ「エンジニアは、なぜ生成AIで仕事が楽にならないのか」(513 users)、HN「AIは簡単な部分をより簡単に、難しい部分をより難しくする」(238pt)、HN「生成するのをやめて、考え始めよう」(99pt) が同日に注目。

### 共通する洞察
- AIが簡単な作業を自動化 → 残る仕事は「難しい判断」ばかり → **体感的な負荷は下がらない**
- 「AIで並列開発に挑んだら、逆に効率を落とした話」(46 users) — コンテキストスイッチの増加がボトルネック
- 「バイブコーディングが脆弱なコード量産」(22 users) — AIに任せきりのリスク

### Deep Work思想との接続
- AIが定型作業を処理する分、**人間は「何を作るか」「なぜ作るか」の判断に集中すべき**
- Agent Teamsはこの文脈で理にかなっている: AIチームに実装を委譲し、人間はアーキテクチャ判断とレビューに専念
- ただし承認ボトルネック問題（Agent Teams実践記事の知見）は、まさにこの「人間の判断待ち」が律速になることを示している
